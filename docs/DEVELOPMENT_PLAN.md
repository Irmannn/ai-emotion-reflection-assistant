# Codex 开发指令

请根据 PRD 开发项目，但必须按阶段推进，不要一次性实现所有功能。

## 总体要求

技术栈：

- 前端：Next.js + React + TypeScript + Tailwind CSS
- 后端：Python + FastAPI
- 数据库：SQLite
- ORM：SQLModel 或 SQLAlchemy
- AI：通过后端调用大模型 API
- API Key：只能放在后端 `.env` 中

## 开发原则

- 每次只完成一个阶段。
- 每个阶段完成后，先告诉用户改了哪些文件。
- 每个阶段必须给出本地运行和测试方法。
- 不要跳过后端存储。
- 不要把 API Key 写进前端代码。
- 不要实现 PRD 之外的复杂功能。
- 代码要尽量清晰，适合初学全栈的人阅读。
- 对关键代码添加简短注释。
- 每个阶段完成后，等待用户确认再继续下一阶段。

## 阶段 1：项目骨架

### 目标

- 创建前端 Next.js 项目结构
- 创建后端 FastAPI 项目结构
- 配置基础 README
- 前后端可以分别启动
- 后端提供一个健康检查接口

### 验收

- 前端页面能打开
- 后端 `GET /health` 返回正常
- README 说明如何启动前后端

## 阶段 2：SQLite 数据库

### 目标

- 接入 SQLite
- 创建 `reflection_records` 表
- 实现数据库初始化
- 实现历史记录的基础 CRUD 接口

### 接口

- `GET /api/reflections`
- `GET /api/reflections/{id}`
- `DELETE /api/reflections/{id}`
- `POST /api/reflections/{id}/feedback`

### 验收

- 可以创建测试数据
- 可以查询列表
- 可以查询详情
- 可以删除记录
- 可以提交反馈

## 阶段 3：前端表单和历史记录

### 目标

- 实现情绪复盘表单
- 前端生成并保存 `session_id`
- 实现历史记录列表
- 实现历史详情展示
- 实现删除记录

### 验收

- 用户可以填写表单
- 用户可以看到历史记录
- 用户可以删除历史记录
- 前端和后端接口能正常联调

## 阶段 4：接入大模型 API

### 目标

- 后端从 `.env` 读取模型 API Key
- 后端封装模型调用函数
- 根据用户输入生成结构化 Markdown 复盘报告
- 暂时可以先非流式返回

### 验收

- 前端提交表单后，后端能返回 AI 复盘报告
- AI 报告结构符合 PRD
- 生成完成后保存到 SQLite

## 阶段 5：实现流式输出

### 目标

- 后端实现流式响应
- 前端逐步展示 AI 输出内容
- 流式完成后保存完整报告到数据库
- 前端显示生成中、完成、失败状态

### 验收

- 用户点击生成后，可以看到内容逐步出现
- 生成完成后历史记录中能看到完整报告
- 网络或模型异常时有错误提示

## 阶段 6：RAG 知识库检索增强

### 目标

- 建立情绪复盘/心理学自助知识库
- 实现文档解析、文本切分和向量化
- 接入向量检索
- 生成 AI 复盘报告时检索相关资料
- 在 AI 报告中展示参考依据或引用片段
- 先完成基础 RAG MVP，Reranker、自动评测和复杂安全策略后置

### 验收

- 可以导入或内置知识库资料
- 用户提交复盘后，后端能检索相关知识片段
- AI 报告能结合检索结果生成
- 前端能展示 AI 使用了哪些资料依据
- 没有命中资料时，系统能正常降级为普通 LLM 复盘

## 阶段 7：Agent / Tool Calling 工作流

### 目标

- 设计可被 AI 调用的后端工具函数
- 实现 Tool Calling 基础流程
- 让 AI 根据用户输入决定是否查询历史记录或复盘详情
- 补充工具调用日志和工具权限边界

### 验收

- AI 能调用至少一个后端工具完成具体任务
- 前端能看到 AI 不只是回答，还能触发实际操作
- 工具调用过程有清晰日志或可观察信息
- 工具调用失败时有明确错误提示

## 阶段 8：有状态流式 Agent

### 目标

- 建立 Agent 会话和消息存储，支持同一会话多轮对话。
- 每轮请求读取当前会话最近若干条消息，作为模型上下文。
- 将 Agent 最终回答改为 SSE 流式输出，并展示工具执行过程。
- 流式结束后保存完整助手消息，关联工具调用日志。
- 失败时保留用户输入并提供手动重试。
- 记录模型名、调用耗时和成功/失败状态，为后续观测打基础。

### 验收

- 同一会话中，Agent 能理解前文并连续回答。
- 用户可看到 Agent 回答逐步生成，以及工具调用状态。
- 刷新页面后能从后端恢复会话消息历史。
- 模型失败时，待发送内容不会丢失，用户可重新尝试。

## 阶段 9：LangChain 组件化

### 目标

- 接入 LangChain 的模型、消息、Prompt、Tool 和 Retriever 抽象。
- 用 `ChatOpenAI` 统一 OpenAI-compatible 模型调用，保留现有业务工具和权限校验。
- 用 `@tool` 等标准抽象封装可调用工具，理解模型工具协议与业务执行的边界。
- 建立后端模型配置层，为主模型、摘要模型和后续备用模型提供统一入口。

### 验收

- Agent 能通过 LangChain 模型与工具抽象完成已有查询能力。
- RAG 检索和 Prompt 构建不依赖前端传入模型配置。
- 更换模型配置时，不需要改动业务路由和前端接口。

## 阶段 10：LangGraph 状态工作流

### 目标

- 用 LangGraph 编排“读取记忆 -> RAG/工具判断 -> 执行 -> 生成 -> 保存”的状态图。
- 引入对话摘要记忆，避免无限传递完整历史消息。
- 增加基础长期记忆：保存经用户确认的偏好或稳定信息，并在相关对话中使用。
- 实现一个 Human-in-the-loop 场景：涉及写入或修改用户数据时先请求确认。

### 验收

- 可查看并解释 Agent 工作流中的节点、分支和循环。
- 长会话仍能保持上下文可用，且不会无限增长输入内容。
- 需要确认的写操作不会在模型决定后直接执行。

## 阶段 11：RAG 与数据层升级

### 目标

- 从 SQLite 迁移到 PostgreSQL，并使用 pgvector 存储和检索向量。
- 增加 Reranker 和小型检索测试集，评估引用质量。
- 优化引用来源、无命中降级和外部资料的 Prompt Injection 基础防护。

### 验收

- 复盘数据、会话数据和向量数据可在 PostgreSQL 中正常保存和查询。
- RAG 回答可展示可追溯的引用来源。
- 能运行固定测试问题，比较检索优化前后的结果。

## 阶段 12：生产化与简历展示

### 目标

- 记录真实 Token 用量、调用成本、模型、耗时和错误类型。
- 实现可重试错误的指数退避、主备模型降级和基础限流。
- 增加结构化日志、Prompt 版本和回归评测。
- 评估后接入 LangSmith：用于 Agent / RAG 调用链 Tracing 与固定测试集评测；接入前确认真实情绪复盘内容的脱敏与第三方数据边界。
- 完成 Docker Compose、GitHub Actions、部署说明、README 截图和简历项目描述。
- 可选练习：WebSocket 长任务进度或用户中止 Agent 任务；正常文本生成继续使用 SSE。

### 验收

- 可定位一次模型、RAG 或工具调用失败的原因。
- 临时网络或上游错误有受控重试或降级路径。
- 新环境可按 README 通过 Docker 启动，并由 CI 完成基础检查。
